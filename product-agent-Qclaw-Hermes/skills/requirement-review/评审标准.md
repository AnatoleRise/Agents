# PRD 评审量化标准

## 一、评分体系总览

### 1.1 维度权重表

| 维度 | 权重 | 满分 | 加权满分 |
|------|------|------|----------|
| 完整性 | 20% | 100 | 20 |
| 可测试性 | 20% | 100 | 20 |
| 一致性 | 15% | 100 | 15 |
| 清晰度 | 15% | 100 | 15 |
| 可行性 | 10% | 100 | 10 |
| 边界场景 | 10% | 100 | 10 |
| 优先级 | 5% | 100 | 5 |
| 合规性 | 5% | 100 | 5 |
| **合计** | **100%** | — | **100** |

### 1.2 计算公式

```
维度得分 = 100 - Σ(问题扣分)（最低 0 分）
加权总分 = Σ(维度得分 × 维度权重)
```

---

## 二、问题严重度分级

| 级别 | 扣分 | 判定标准 | 处理要求 |
|------|------|---------|---------|
| 🔴严重 | -25 分/个 | 阻碍开发或导致重大误解的缺陷 | 必须修改后才能进入评审 |
| 🟡警告 | -10 分/个 | 可能导致理解偏差或遗漏 | 建议修改后进入开发 |
| 💡建议 | -3 分/个 | 改进空间，不影响当前开发 | 可在迭代中优化 |

## 三、严重度判定矩阵

### 3.1 🔴严重 问题判定标准

出现以下任一情况，标记为严重：

| 序号 | 情形 | 示例 |
|------|------|------|
| S1 | 缺少产品背景与业务目标 | 文档直接进入技术细节，未说明为什么做、解决什么痛点 |
| S2 | P0（核心功能）缺少验收标准 | 登录注册功能未定义成功标准和异常流程 |
| S3 | 前后描述自相矛盾 | A 章节说支持 iOS，B 章节说仅支持 Android |
| S4 | 依赖未就绪的外部系统/接口且无替代方案 | 依赖第三方 API 但未定义降级策略 |
| S5 | 缺少安全边界条件定义 | 未定义权限校验、越权防护 |
| S6 | 需求描述模糊到无法直接编码 | 「优化用户体验」无具体指标 |
| S7 | 核心触发/路由规则边界未定义 | 如何判断意图归属的条件缺失 |
| S8 | 兜底策略/失败处理缺失 | 大模型兜底处理无输入输出协议 |

### 3.2 🟡警告 问题判定标准

出现以下任一情况，标记为警告：

| 序号 | 情形 | 示例 |
|------|------|------|
| W1 | 同一概念有 2 种及以上不同叫法 | 同一功能在 A 处叫「用户」，B 处叫「客户」 |
| W2 | 非 P0 功能缺少验收标准 | 次要功能无量化指标 |
| W3 | 存在隐性假设 | 「和之前一样」「大家都知道」 |
| W4 | 未覆盖边界场景 | 空值、超长输入、网络异常无处理定义 |
| W5 | 优先级标注不合理 | 所有功能都是 P0，或未标注优先级 |
| W6 | 包含 NESMA 规避敏感词 | 使用「是否」「逻辑」「判断」等词汇 |
| W7 | 后台配置/管理方案缺失 | 「由后台配置」但无配置入口定义 |
| W8 | 排期延误未更新方案 | 引用已过期的上线日期 |

### 3.3 💡建议 问题判定标准

出现以下任一情况，标记为建议：

| 序号 | 情形 | 示例 |
|------|------|------|
| T1 | 表述可更精确 | 「一段时间」可改为「近 7 天」 |
| T2 | 非关键信息可补充 | 建议增加版本变更记录 |
| T3 | 格式/排版优化 | 建议统一表格格式 |
| T4 | 措辞可优化 | 「无法实现」建议改为「本期暂不支持」 |

## 四、维度评分细则

### 4.1 完整性（权重 20%）

| 检查项 | 权重分 | 缺失扣分 | 部分缺失扣分 |
|--------|--------|---------|-------------|
| 产品背景与业务目标 | 20 | -25（严重） | -10（警告） |
| 目标用户/用户画像 | 15 | -25（严重） | -10（警告） |
| 功能需求覆盖业务场景 | 15 | -10（警告） | -3（建议） |
| 非功能需求（性能/安全/兼容/可用） | 15 | -10（警告） | -3（建议） |
| 验收标准/成功指标 | 15 | -25（严重） | -10（警告） |
| 优先级标注（P0/P1/P2） | 5 | -10（警告） | -3（建议） |
| 数据模型/业务流程图 | 10 | -10（警告） | -3（建议） |
| 版本变更记录 | 5 | — | -3（建议） |

### 4.2 一致性（权重 15%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 术语统一 | 同一概念是否有多重叫法 | 1 处 -10，2 处以上 -25 |
| 功能描述与数据模型一致 | 功能描述中的字段是否在数据模型中定义 | 每项 -10 |
| 用户故事与功能列表对应 | 用户故事是否有对应的功能实现 | 每项 -10 |
| 前后端描述匹配 | 前端页面描述与后端接口定义是否匹配 | 每项 -10 |
| 流程描述一致 | 不同章节对同一流程的描述是否一致 | 每项 -25 |
| 枚举值全局统一 | 版本号、状态枚举值等是否统一 | 每项 -10 |

### 4.3 可测试性（权重 20%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 量化验收标准 | 每个功能是否有可量化的验收标准 | 每项 -25 |
| 性能指标数值化 | 性能需求是否有具体数值（如响应时间 < 2s） | 每项 -10 |
| 边界条件定义 | 最大值、最小值、空值处理 | 每项 -10 |
| 异常流程预期行为 | 异常输入是否有预期行为描述 | 每项 -10 |
| 状态机完整性 | 所有状态转换是否定义完整 | 缺失状态 -25 |
| 可编写测试用例 | 需求是否能用 Given-When-Then 格式编写测试 | 无法编写 -25 |

### 4.4 可行性（权重 10%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 技术方案已知 | 是否有已知实现方案或技术预研 | -25 |
| 外部依赖就绪 | 是否依赖未就绪的系统/接口 | -25 |
| 时间规划合理 | 排期是否与功能复杂度匹配 | -10 |
| 技术风险识别 | 是否有未识别的技术风险 | -10 |
| 合规/法律风险评估 | 是否评估了合规风险 | -10 |
| 替代方案定义 | 无法实现的能力是否有降级/替代方案 | 每项 -10 |

### 4.5 清晰度（权重 15%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 避免模糊词汇 | 是否使用「优化」「提升」「改善」等词 | 每项 -10 |
| 责任方明确 | 每个需求是否有明确的责任方 | 每项 -10 |
| 无隐性假设 | 是否避免「大家都知道」「和之前一样」 | 每项 -10 |
| 流程图可读 | 流程图/时序图是否清晰 | 缺失 -10 |
| 可直接编码 | 描述是否足够具体让开发人员直接编码 | -25 |
| 配置/入口定义 | 「后台配置」等是否有入口说明 | 每项 -10 |

### 4.6 优先级（权重 5%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 标注优先级 | 是否标注了每个功能点的优先级 | 未标注 -10 |
| P0 范围合理 | P0 是否不超过总量的 30% | 超标 -10 |
| 优先级与业务价值匹配 | 高投入低回报功能是否被识别 | -10 |
| 无伪优先级现象 | 是否存在所有功能都是高优先级 | -10 |
| 分期/分版本规划 | 是否有 V1/V2 分期规划 | 缺失 -10 |

### 4.7 边界场景（权重 10%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| 异常输入处理 | 异常输入的处理方式是否定义 | 每项 -10 |
| 网络异常/超时 | 网络异常/超时/重试策略是否描述 | 每项 -10 |
| 并发场景 | 多人同时操作是否有定义 | 缺失 -10 |
| 数据量极端增长 | 数据量增长时的表现是否有预期 | 缺失 -10 |
| 安全边界 | 权限越界、未授权访问等是否考虑 | 缺失 -25 |
| 国际化/多语言 | 国际化/多时区场景是否覆盖 | 缺失 -3 |
| 离线场景 | 离线场景是否有定义（如适用） | 缺失 -3 |

### 4.8 合规性（权重 5%）

| 检查项 | 检查内容 | 扣分 |
|--------|---------|------|
| NESMA 敏感词规避 | 是否规避「是否」「逻辑」「判断」「解析」等词 | 每处 -3 |
| 动宾结构 | 功能点描述是否采用动宾结构 | 每处 -3 |
| 非功能需求占比 | 非功能需求是否不超过 40% | 超标 -10 |
| 功能唯一性 | 是否存在重复功能项 | 每项 -10 |
| 行业/公司规范 | 是否符合公司/行业规范要求 | 缺失 -10 |

## 五、评级标准

| 等级 | 分数区间 | 含义 | 处理建议 |
|------|---------|------|---------|
| A | 90-100 | 优秀 | 可直接进入开发阶段 |
| B | 75-89 | 良好 | 修改警告项后可进入开发 |
| C | 60-74 | 及格 | 修改严重项后重新评审 |
| D | 0-59 | 不合格 | 需重大修改后重新提交评审 |

### 5.1 评级与严重度关联规则

| 评级 | 允许的严重问题数 | 允许的警告问题数 | 建议 |
|------|----------------|----------------|------|
| A | 0 | ≤ 2 | 直接进入开发 |
| B | 0 | 3-5 | 修改警告项后进入开发 |
| C | 1-2 | ≤ 8 | 修改严重项后重新评审 |
| D | ≥ 3 | 不限 | 重大修改后重新提交 |

> 注：即使加权总分达到 B 级，若严重问题数 ≥ 3，评级强制降为 D。

## 六、NESMA 功能点评估规范

### 6.1 敏感词规避清单

以下词汇在功能描述中禁止使用，应替换为动宾结构：

| 禁用词 | 建议替换 |
|--------|---------|
| 是否 | 判断 → 检查/确认/验证 |
| 是不是 | 同上 |
| 逻辑 | 逻辑处理 → 处理规则/业务流程 |
| 校验 | 校验 → 检查/验证/核对 |
| 判断 | 同上 |
| 判定 | 同上 |
| 解析 | 解析 → 分析/提取/识别 |
| 逻辑处理 | 同上 |
| 优化 | 优化 → 提升/改进 + 量化指标 |
| 改善 | 同上 |

### 6.2 功能点描述规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 动宾结构 | 功能描述必须以动词 + 宾语形式 | ✅「创建用户账号」❌「用户账号是否创建」 |
| 原子性 | 每个功能点只描述一个功能 | ✅「修改用户密码」「重置用户密码」分别描述 |
| 唯一性 | 不允许出现重复功能项 | 检查全文无重复功能描述 |
| 量化非功能 | 非功能需求占比不超过 40% | 性能、安全等非功能需求需量化 |

## 七、角色评审量化标准

### 7.1 默认三角色

#### 7.1.1 专家产品经理

**评审重点：产品战略、商业价值、用户价值、市场定位、需求合理性**

| 序号 | 检查项 | 严重 | 警告 | 建议 |
|------|--------|------|------|------|
| PM-1 | 产品背景是否说明业务痛点和市场机会（为什么做） | 缺失 | 描述不充分 | 可补充数据支撑 |
| PM-2 | 目标用户群体是否明确（用户画像、使用场景、用户旅程） | 缺失 | 画像不完整 | 可补充用户旅程图 |
| PM-3 | 需求是否对齐产品愿景和长期战略 | 孤立功能无愿景支撑 | 对齐关系模糊 | 可补充愿景声明 |
| PM-4 | 功能优先级是否与商业价值匹配 | 高投入低回报功能被标为 P0 | 优先级依据不充分 | 可补充 ROI 分析 |
| PM-5 | 是否有竞品分析或市场对标 | 无竞品分析 | 竞品分析不深入 | 可补充差异化矩阵 |
| PM-6 | MVP 范围是否合理 | MVP 过大（> 30% 功能） | MVP 边界模糊 | 可明确 MVP 核心假设 |
| PM-7 | 成功指标是否定义（DAU/留存率/转化率等业务指标） | 缺失 | 仅技术指标无业务指标 | 可补充北极星指标 |
| PM-8 | 用户故事是否完整（As a XX, I want XX, so that XX） | 缺失 | 格式不规范 | 可补充验收场景 |
| PM-9 | 是否存在「伪需求」（过度设计） | 过度设计与核心需求无关 | 部分功能可裁剪 | 可补充需求验证依据 |
| PM-10 | 功能间依赖关系和协同效应是否说明 | 关键依赖未识别 | 依赖关系描述不清 | 可补充依赖关系图 |
| PM-11 | 是否有数据驱动的决策依据 | 无任何用户调研/数据支撑 | 数据支撑不充分 | 可补充调研样本量 |
| PM-12 | 需求变更/演进路径是否考虑（V1/V2 规划） | 无演进规划 | V2 方向不明确 | 可补充路线图 |
| PM-13 | 是否考虑用户增长和留存策略 | 缺失 | 仅考虑拉新无留存 | 可补充 AARRR 模型 |
| PM-14 | 付费模式/商业模式是否定义（如适用） | 缺失 | 模式不清晰 | 可补充定价策略 |

**评分规则**：每个 🔴严重 -25 分，🟡警告 -10 分，💡建议 -3 分，满分 100，最低 0。

#### 7.1.2 项目经理

**评审重点：项目排期、资源分配、风险管理、交付标准、协作机制**

| 序号 | 检查项 | 严重 | 警告 | 建议 |
|------|--------|------|------|------|
| PJ-1 | 项目范围是否明确且可控 | 范围不明确 | 存在范围蔓延风险 | 可补充范围边界说明 |
| PJ-2 | 功能拆分是否合理（可按阶段/模块分批交付） | 不可拆分 | 拆分粒度不合理 | 可补充拆分策略 |
| PJ-3 | 交付里程碑是否定义 | 缺失 | 交付物/验收标准不明确 | 可补充里程碑表 |
| PJ-4 | 资源需求是否评估（人力/时间/技术资源） | 未评估 | 评估不充分 | 可补充资源计划 |
| PJ-5 | 关键路径是否识别 | 关键路径未识别 | 关键路径描述不清 | 可补充关键路径图 |
| PJ-6 | 依赖关系是否明确（内部/外部/第三方） | 关键依赖未识别 | 依赖影响未评估 | 可补充依赖矩阵 |
| PJ-7 | 技术风险是否评估并有应对预案 | 高风险无预案 | 预案不具体 | 可补充风险登记表 |
| PJ-8 | 沟通机制是否定义（站会/周报/评审节点） | 缺失 | 频率/形式不明确 | 可补充沟通计划 |
| PJ-9 | 变更管理流程是否说明 | 缺失 | 流程不完整 | 可补充变更审批表 |
| PJ-10 | 验收标准是否全局统一（DoD） | 无 DoD 定义 | DoD 不统一 | 可补充 DoD 清单 |
| PJ-11 | 回滚/应急预案是否定义 | 缺失 | 预案不具体 | 可补充回滚步骤 |
| PJ-12 | 多端/多团队并行开发协调策略 | 缺失 | 协调机制不完整 | 可补充协作流程 |
| PJ-13 | 上线发布策略是否定义（灰度/全量/回滚窗口） | 缺失 | 策略不完整 | 可补充发布计划 |

#### 7.1.3 测试工程师

**评审重点：可测试性、测试覆盖率、异常场景**

| 序号 | 检查项 | 严重 | 警告 | 建议 |
|------|--------|------|------|------|
| T-1 | 每个功能点是否能编写 Given-When-Then 测试用例 | 无法编写 | 用例不完整 | 可补充边界用例 |
| T-2 | 是否定义了验收测试通过/失败标准 | 缺失 | 标准不明确 | 可补充验收 checklist |
| T-3 | 异常流程是否覆盖（空值/超长/特殊字符/网络断开/超时） | 关键异常未覆盖 | 部分异常未覆盖 | 可补充异常清单 |
| T-4 | 边界值是否明确（最大值/最小值/临界值） | 缺失 | 不完整 | 可补充边界值表 |
| T-5 | 状态机所有状态转换是否定义完整 | 遗漏关键状态 | 转换条件不完整 | 可补充状态转换图 |
| T-6 | 是否有端到端测试场景描述 | 缺失 | 场景不完整 | 可补充 E2E 流程图 |
| T-7 | 性能指标是否有可量化测试目标（QPS/响应时间/并发数） | 缺失 | 指标不完整 | 可补充性能测试方案 |
| T-8 | 是否有回归测试范围说明 | 缺失 | 范围不完整 | 可补充影响分析矩阵 |
| T-9 | 多端一致性是否有测试要求 | 缺失 | 要求不明确 | 可补充一致性测试清单 |
| T-10 | 数据迁移/兼容升级是否有测试方案 | 缺失 | 方案不完整 | 可补充迁移测试计划 |

### 7.2 按需激活角色

#### 7.2.1 UI/UX 设计师

**触发条件**：PRD 含页面设计、交互流程、视觉风格、用户体验、动效、响应式、组件库相关内容

## 八、角色评分汇总规则

### 8.1 角色激活与评分关联

| 角色类型 | 激活方式 | 评分权重 | 说明 |
|---------|---------|---------|------|
| 专家产品经理 | 默认激活 | 纳入综合评分 | 所有 PRD 必评 |
| 项目经理 | 默认激活 | 纳入综合评分 | 所有 PRD 必评 |
| 测试工程师 | 默认激活 | 纳入综合评分 | 所有 PRD 必评 |
| 其他角色 | 按关键词自动激活 | 纳入综合评分 | 仅当 PRD 涉及相关内容时激活 |
| 未激活角色 | 不参与评审 | 不纳入评分 | 输出「未激活」说明 |

### 8.2 角色评分计算

```
角色得分 = 100 - Σ(该角色检查项扣分)
综合评分 = Σ(通用维度加权分 × 0.7) + Σ(角色得分 × 0.3 / 激活角色数)
```

> 通用评审（8 大维度）权重占 70%，角色评审权重占 30%。当仅激活默认三角色时，角色部分均分给三个角色。

### 8.3 角色问题去重规则

当多个角色发现同一问题时：
1. 按问题内容去重，保留最严重的严重度
2. 在报告中注明涉及的角色视角
3. 扣分仅计算一次，不重复扣分

## 九、快速评分卡

用于快速扫描模式下的简版评分：

| 维度 | 得分 (0-100) | 主要问题数 |
|------|-------------|-----------|
| 完整性 | /100 | |
| 一致性 | /100 | |
| 可测试性 | /100 | |
| 清晰度 | /100 | |

**快速评分总分** = (完整性 + 一致性 + 可测试性 + 清晰度) / 4

| 快速评分等级 | 分数 | 建议 |
|-------------|------|------|
| 通过 | ≥ 70 | 可进入开发 |
| 有条件通过 | 55-69 | 修改关键问题后进入开发 |
| 不通过 | < 55 | 需要深度评审后重新评审 |
