# PRD 质量判定标准

## 评分维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| 完整性 | 20% | PRD 是否包含所有必要章节和信息 |
| 一致性 | 15% | 术语、数据定义、流程描述是否前后一致 |
| 可测试性 | 20% | 每个需求是否有可量化的验收标准 |
| 可行性 | 10% | 技术实现是否可行，依赖是否就绪 |
| 清晰度 | 15% | 需求描述是否具体明确，无歧义 |
| 优先级 | 5% | 优先级标注是否合理，MVP 范围是否清晰 |
| 边界场景 | 10% | 异常流程、安全边界、极端情况是否覆盖 |
| 合规性 | 5% | 是否符合规范和标准要求 |

## 单维度评分规则

每个维度满分 100 分，按发现的问题扣分：

| 严重程度 | 单次扣分 | 说明 |
|---------|---------|------|
| 🔴严重 | -25 分 | 致命缺陷，阻碍开发或导致重大误解 |
| 🟡警告 | -10 分 | 可能导致理解偏差或遗漏 |
| 💡建议 | -3 分 | 改进空间，不影响当前开发 |

最低 0 分，不出现负分。

## 严重程度判定指南

### 🔴严重（致命）

满足以下任一条件即判定为 🔴严重：
- 缺少产品背景或目标描述，无法理解业务价值
- P0 功能缺少验收标准，无法验证是否实现
- 功能描述存在自相矛盾，前后不一致
- 依赖外部系统但该系统的接口/数据格式未定义
- 安全相关需求缺少边界条件（如权限校验、数据加密）
- 存在模糊需求导致开发人员无法直接编码

### 🟡警告（警告）

满足以下任一条件即判定为 🟡警告：
- 核心术语不统一（同一概念有 2 种以上叫法）
- 非 P0 功能缺少验收标准
- 存在隐性假设（"和之前一样"、"大家都知道"）
- 边界场景未覆盖但可能影响用户体验
- 优先级标注不合理（所有功能都是 P0）
- 使用了 NESMA 敏感词但未影响理解

### 💡建议（建议）

满足以下任一条件即判定为 💡建议：
- 描述可以更精确但不影响当前理解
- 缺少非关键性的补充说明
- 格式/排版建议
- 可以优化的措辞

## 总体评级标准

| 等级 | 分数范围 | 含义 | 行动建议 |
|------|---------|------|---------|
| A | 90-100 | 优秀 | 可直接进入开发阶段 |
| B | 75-89 | 良好 | 修改 🟡警告 项后可进入开发 |
| C | 60-74 | 及格 | 需修改关键问题后重新评审 |
| D | 0-59 | 不合格 | 需重大修改后重新提交评审 |

## NESMA 规范要点

### 必须遵守

1. **动宾结构**：功能描述必须以动词开头（新增、修改、删除、查询、展示、提交等）
2. **规避敏感词**：不使用"是否、是不是、逻辑、校验、判断、判定、解析、逻辑处理、优化"
3. **原子化**：每个功能点是业务上不可再分的最小操作单元
4. **差异化**：不同功能的 FTR（引用文件）和 DET（数据元素）应有明显差异
5. **非功能占比**：纯非功能需求不超过总功能点的 40%

### 常见问题

| 错误写法 | 正确写法 |
|---------|---------|
| 用户信息查询 | 建设用户信息查询功能 |
| 优化列表加载速度 | 将列表接口响应时间从 5s 优化至 2s 以内 |
| 数据校验 | 建设数据格式校验能力 |
| 逻辑处理 | 建设 XXX 业务处理流程 |
| 判断用户权限 | 建设用户权限判定功能 |
