# PRD 评审检查清单

> 本清单分为两组：快速扫描（Q1-Q12）和深度评审（Q1-Q24）。深度评审包含所有快速扫描检查项。

---

## 快速扫描检查项（12 项）

### 完整性

- [ ] **Q1**: PRD 是否包含产品背景与目标说明？缺少背景文档会让开发不理解业务价值。
- [ ] **Q2**: 是否定义了目标用户或用户画像？
- [ ] **Q3**: 功能列表是否完整？是否每个功能都有描述？
- [ ] **Q4**: 是否标注了功能优先级（P0/P1/P2 或 Must/Should/Could）？

### 一致性

- [ ] **Q5**: 全文核心术语是否统一？（如同一功能在不同章节叫法不同）
- [ ] **Q6**: 功能描述与验收标准是否对应？（有功能无验收标准 = 不可测试）

### 可测试性

- [ ] **Q7**: 每个 P0/P1 功能是否有可量化的验收标准？（如"响应时间 < 2s"而非"响应速度快"）
- [ ] **Q8**: 是否定义了边界值？（最大值、最小值、空值、超限值的处理）

### 清晰度

- [ ] **Q9**: 是否存在模糊词汇？（优化、提升、改善、增强、友好、流畅 —— 这些词无法量化）
- [ ] **Q10**: 是否存在隐性假设？（"和之前一样"、"大家都知道"、"用户自然会..."）
- [ ] **Q11**: 需求描述是否足够具体，让开发人员可以直接编码而无需猜测？

### 快速评分

- [ ] **Q12**: 统计 🔴严重 问题数量。如果 ≥ 3 个，直接判定为"不建议进入开发"。

---

## 深度评审额外检查项（13 项）

### 可行性

- [ ] **Q13**: 技术方案是否可行？是否有已知技术方案或需要技术预研？
- [ ] **Q14**: 是否依赖未就绪的外部系统、第三方 API 或数据源？
- [ ] **Q15**: 是否有未识别的技术风险？（如性能瓶颈、数据迁移难度）

### 边界场景

- [ ] **Q16**: 异常输入的处理方式是否定义？（非法字符、超长文本、特殊格式）
- [ ] **Q17**: 网络异常/超时/重试策略是否描述？
- [ ] **Q18]: 并发场景是否有定义？（多人同时操作同一资源）
- [ ] **Q19**: 安全边界是否考虑？（权限越界、未授权访问、SQL 注入、XSS）

### 优先级

- [ ] **Q20**: P0（MVP）范围是否合理？（不超过总量的 30%）
- [ ] **Q21]: 是否存在"所有功能都是高优先级"的伪优先级现象？

### 合规性

- [ ] **Q22**: 功能点描述是否规避了 NESMA 敏感词？（是否、是不是、逻辑、判断、解析、优化）
- [ ] **Q23**: 功能点描述是否采用动宾结构？（如"建设用户查询功能"而非"用户查询"）
- [ ] **Q24**: 功能描述是否具有唯一性？（无重复或高度相似的功能项）
