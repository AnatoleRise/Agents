# 常见 PRD 问题模式库

> 本库收录 30+ 种 PRD 中高频出现的问题模式，每种模式包含：模式名称、反例、问题分析、改写示例。

---

## 一、模糊词汇类

### P-01: 模糊性能描述
- **反例**：「系统响应速度快」
- **问题**：「快」是主观感受，开发人员无法据此确定性能目标
- **改写**：「系统 API 接口 P95 响应时间 < 500ms，页面首屏加载时间 < 2s」

### P-02: 模糊优化描述
- **反例**：「优化用户体验」
- **问题**：无法量化，无法验证是否达成
- **改写**：「将用户完成核心操作的步骤从 5 步减少到 3 步」

### P-03: 模糊质量描述
- **反例**：「提升系统稳定性」
- **问题**：稳定性缺少明确定义
- **改写**：「系统可用性从 99.5% 提升至 99.9%，月度故障次数从 5 次降低至 1 次以内」

### P-04: 模糊安全描述
- **反例**：「加强安全防护」
- **问题**：缺少具体的安全措施
- **改写**：「所有 API 接口启用 JWT 鉴权，敏感数据传输使用 AES-256 加密，密码存储使用 bcrypt」

### P-05: 模糊扩展性描述
- **反例**：「系统具有良好的扩展性」
- **问题**：扩展性是架构属性，不是功能需求
- **改写**：「采用插件化架构，新增功能模块无需修改核心代码」

---

## 二、结构缺失类

### P-06: 缺少验收标准
- **反例**：「支持文件上传功能」
- **问题**：没有说明什么算「支持」，文件大小、格式、数量等约束未定义
- **改写**：「支持文件上传功能，单次上传不超过 10MB，支持 jpg/png/pdf 格式，单次最多上传 5 个文件，上传进度条实时显示」

### P-07: 缺少异常流程
- **反例**：「用户提交订单后跳转至支付页面」
- **问题**：只描述了正常流程，未描述异常情况
- **改写**：「用户提交订单后：① 成功时跳转支付页面；② 库存不足时弹窗提示并返回购物车；③ 网络超时时提示重试并保留草稿」

### P-08: 缺少边界条件
- **反例**：「输入年龄」
- **问题**：未定义合法范围
- **改写**：「输入年龄，仅允许 1-150 的整数，超出范围时提示「请输入有效年龄（1-150）」」

### P-09: 缺少权限定义
- **反例**：「管理员可以删除用户」
- **问题**：未定义「管理员」的具体角色，也未定义删除是软删除还是硬删除
- **改写**：「系统管理员（role=admin）可以删除普通用户（role=user），删除为软删除（is_deleted=true），被删除用户 30 天内可恢复」

### P-10: 缺少版本兼容说明
- **反例**：「APP 支持推送通知」
- **问题**：未说明支持的 OS 版本
- **改写**：「APP 支持推送通知，iOS 14+ 使用 APNs，Android 8.0+ 使用 FCM，低版本设备不支持」

## 三、不一致类

### P-11: 术语不一致
- **反例**：第一章叫「用户」，第三章叫「客户」，第五章叫「会员」
- **问题**：同一概念多个名称，开发人员困惑
- **改写**：全文统一使用「用户」，在术语表中明确定义

### P-12: 状态枚举不一致
- **反例**：功能 A 定义订单状态为 [待支付, 已支付, 已完成]，功能 B 定义为 [pending, paid, shipped, completed]
- **问题**：同一实体在不同功能中的状态定义不一致
- **改写**：统一定义订单状态枚举并在全文引用

### P-13: 数据流向不闭环
- **反例**：功能 A 说「将数据存储到数据库 X」，功能 B 说「从数据库 Y 读取数据」
- **问题**：数据来源和去向不匹配
- **改写**：明确数据从功能 A 写入的表与功能 B 读取的表之间的映射关系

### P-14: 前后数字矛盾
- **反例**：背景说「日活 10 万」，性能需求说「支持 100 万并发」
- **问题**：两个数字量级差距过大，其中一个可能有误
- **改写**：核实并统一数据基准

## 四、优先级类

### P-15: 全高优先级
- **反例**：30 个功能点中 28 个标注为 P0
- **问题**：P0 失去了意义，无法确定 MVP 范围
- **改写**：重新评估，P0 不超过总量的 30%

### P-16: 优先级与业务价值不匹配
- **反例**：核心支付流程是 P2，Logo 更换是 P0
- **问题**：优先级标注不合理
- **改写**：按业务价值重新评估优先级

### P-17: 缺少优先级标注
- **反例**：功能列表中没有优先级字段
- **问题**：无法区分 MVP 和后续迭代
- **改写**：为每个功能点标注 P0/P1/P2

## 五、NESMA 合规类

### P-18: 非动宾结构
- **反例**：「用户查询」
- **改写**：「建设用户信息查询功能」

### P-19: 使用敏感词「是否」
- **反例**：「判断用户是否为 VIP」
- **改写**：「建设用户身份判定功能」

### P-20: 使用敏感词「逻辑」
- **反例**：「订单分配逻辑」
- **改写**：「建设订单自动分配功能」

### P-21: 使用敏感词「优化」
- **反例**：「优化搜索算法」
- **改写**：「建设搜索结果排序功能，支持按相关度、时间、热度排序」

### P-22: 使用敏感词「校验」
- **反例**：「数据格式校验」
- **改写**：「建设数据格式验证功能」

### P-23: 使用敏感词「解析」
- **反例**：「JSON 数据解析」
- **改写**：「建设 JSON 数据接收与处理能力」

### P-24: 使用敏感词「处理」
- **反例**：「异常处理」
- **改写**：「建设异常记录与通知功能」

## 六、可行性类

### P-25: 依赖未就绪
- **反例**：「对接第三方物流 API」，但未提供 API 文档或确认接口可用
- **问题**：开发可能因依赖未就绪而阻塞
- **改写**：在 PRD 中标注依赖状态，并给出降级方案（如使用 Mock 数据开发）

### P-26: 技术方案未验证
- **反例**：「使用 WebRTC 实现实时视频通话」（团队从未做过 WebRTC）
- **问题**：技术方案存在不确定性
- **改写**：标注「需要技术预研」，给出预研时间预估

### P-27: 数据迁移未评估
- **反例**：「将用户表从 MySQL 迁移到 MongoDB」
- **问题**：未评估数据量、迁移时间、停机窗口
- **改写**：补充数据迁移方案，包括数据量、预计耗时、回滚策略

---

## 七、边界场景类

### P-28: 未定义并发行为
- **反例**：「用户点击提交按钮后创建订单」
- **问题**：用户快速点击两次会发生什么？
- **改写**：「用户点击提交按钮后，按钮变为禁用状态 3 秒，防止重复提交」

### P-29: 未定义空状态
- **反例**：「展示订单列表」
- **问题**：用户没有订单时展示什么？
- **改写**：「展示订单列表，无订单时展示空状态页面，包含引导文案和创建订单按钮」

### P-30: 未定义超长内容处理
- **反例**：「展示用户昵称」
- **问题**：昵称超长时是否截断？换行？
- **改写**：「展示用户昵称，超过 20 个字符时截断并显示省略号」

### P-31: 未定义网络异常
- **反例**：「上传图片」
- **问题**：上传过程中网络断开怎么办？
- **改写**：「上传图片，支持断点续传，网络断开时保存进度，恢复后自动继续上传」

## 八、其他常见问题

### P-32: 隐性假设
- **反例**：「和现有流程保持一致」
- **问题**：读者不知道「现有流程」是什么
- **改写**：明确描述现有流程或引用相关文档

### P-33: 自然语言歧义
- **反例**：「用户可以删除或修改自己的评论」
- **问题**：是「（删除或修改）自己的评论」还是「删除（自己或别人的）评论」？
- **改写**：「用户可以删除自己的评论，也可以修改自己已发布的评论」

### P-34: 缺少触发条件
- **反例**：「发送短信验证码」
- **问题**：什么情况下触发？
- **改写**：「用户点击「获取验证码」按钮且手机号格式正确时，发送 6 位数字短信验证码，60 秒内不可重复发送」

### P-35: 功能描述重复
- **反例**：功能 A「用户信息修改」和功能 B「个人资料更新」实际是同一件事
- **问题**：导致功能点重复计数
- **改写**：合并为单一功能描述
