# 问题跟踪工作流

> 本文件定义问题工单的状态机、流转规则、修复验证检查清单和整改 SLA，供 Issue Tracker Skill 在执行跟踪任务时作为操作依据。

---

## 一、问题状态机

```
                    ┌──────────┐
                    │ 待处理    │  ← 初始状态 / 驳回后回退
                    │ pending  │
                    └────┬─────┘
                         │ 责任人认领 / 进入修复
                         ▼
                    ┌──────────┐
                    │ 处理中    │
                    │ in_progress│
                    └────┬─────┘
                         │ 新版 PRD 发布，宣称修复
                         ▼
                    ┌──────────┐
                    │ 已修复    │
                    │  fixed   │
                    └────┬─────┘
                         │
              ┌──────────┴──────────┐
              │ 复检                │
              ▼                     ▼
         ┌──────────┐         ┌──────────┐
         │ 已验证    │         │ 驳回      │
         │ verified │         │ rejected │
         └────┬─────┘         └────┬─────┘
              │                    │
              │ 终态               │ 修复不充分，回退
              │                    ▼
              │               ┌──────────┐
              │               │ 待处理    │
              │               │ pending  │  ← 重新进入队列
              │               └────┬─────┘
              │                    │ 产品经理确认驳回合理
              │                    ▼
              │               ┌──────────┐
              │               │ 已关闭    │
              │               │ closed   │  ← 终态（不再修复）
              │               └──────────┘
              │
              ▼
         终态（问题已解决）
```

### 状态枚举定义

| 状态 | 英文标识 | 说明 | 是否终态 |
|------|---------|------|---------|
| 待处理 | `pending` | 工单创建初始状态，或驳回后回退 | 否 |
| 处理中 | `in_progress` | 已被责任人认领，修复进行中 | 否 |
| 已修复 | `fixed` | 新版 PRD 发布，宣称该问题已修复 | 否 |
| 已验证 | `verified` | 复检通过，修复充分，问题已关闭 | 是 |
| 驳回 | `rejected` | 复检不通过，修复不充分或引入新问题 | 否（瞬态，立即流转） |
| 已关闭 | `closed` | 产品经理确认驳回合理，不再修复（如：建议类问题延后到下个版本） | 是 |

## 二、状态流转规则

### 2.1 `pending → in_progress`

**触发条件**：
- 工单的责任人已被分配（`assignee` 不为空）
- 产品经理或项目经理确认该工单进入本轮修复范围

**执行操作**：
- `status = in_progress`
- `updated_at = now()`

**注意事项**：
- 致命问题的工单应在创建后 4 小时内进入 `in_progress`
- 如超时未认领，在报告中触发 SLA 预警

### 2.2 `in_progress → fixed`

**触发条件**：
- 新版 PRD 已发布
- 工单的原文定位出现在新版 PRD 的变更章节中
- 变更内容被 Diff 引擎识别为「正向匹配」

**执行操作**：
- `status = fixed`
- `updated_at = now()`

**注意事项**：
- 仅凭「章节被修改」不自动判定为 `fixed`。需要产品经理或项目经理确认「该修改意图确实是修复此问题」
- 如产品经理标注为「已修复」但 Diff 中未检测到变更，标记为异常，需人工确认

### 2.3 `fixed → verified`

**触发条件**：
- 复检通过。判定标准见本章「复检判定规则」

**执行操作**：
- `status = verified`
- `resolved_at = now()`（首次验证通过的时间）
- `updated_at = now()`

### 2.4 `fixed → rejected`

**触发条件**：
- 复检不通过。判定标准见本章「复检判定规则」

**执行操作**：
- `status = rejected`（瞬态）
- `fix_iterations += 1`
- `updated_at = now()`
- 然后立即流转至 `pending`

**注意事项**：
- 驳回时应在报告中附驳回原因和具体复检发现
- 如 `fix_iterations >= 3`，触发升级预警，建议提升问题优先级或重新评估修复方案

### 2.5 `rejected → pending`

**触发条件**：
- 驳回后自动回退

**执行操作**：
- `status = pending`
- 保留 `fix_iterations` 和驳回记录

### 2.6 `rejected → closed`（或 `pending → closed`）

**触发条件**（满足任一）：
- 产品经理确认该问题不修复（如：建议类问题延后到下个版本）
- 问题被判定为「现有条件下无法修复」且有文档记录
- 原始需求变更导致该问题不再适用

**执行操作**：
- `status = closed`
- `closed_at = now()`
- `updated_at = now()`

**注意事项**：
- 关闭时必须附关闭原因
- 致命问题不允许关闭。如确需关闭，必须升级审批
- 已关闭的问题如果后续版本重新出现同类问题，创建新工单而非重新打开

## 三、修复验证检查清单

当对 `fixed` 状态的工单执行复检时，按以下规则映射检查项：

### 3.1 问题维度 → 检查项映射表

| 问题所在维度 | 对应检查项（Q1-Q24） | 复检重点 |
|-------------|---------------------|---------|
| 完整性 | Q1, Q2, Q3, Q4 | 检查新版 PRD 对应章节是否补全了缺失的信息 |
| 一致性 | Q5, Q6 | 检查术语是否统一、功能与验收标准是否对应 |
| 可测试性 | Q7, Q8 | 检查是否新增了量化验收标准、边界值定义 |
| 清晰度 | Q9, Q10, Q11 | 检查是否消除了模糊词汇和隐性假设 |
| 可行性 | Q13, Q14, Q15 | 检查是否明确了技术方案、依赖状态 |
| 边界场景 | Q16, Q17, Q18, Q19 | 检查是否覆盖了异常输入、并发、安全边界 |
| 优先级 | Q20, Q21 | 检查优先级标注是否合理 |
| 合规性 | Q22, Q23, Q24 | 检查 NESMA 合规、动宾结构、功能唯一性 |

> 完整检查项定义参见 `requirement-review/references/review-checklist.md`

### 3.2 问题模式 → 复检重点映射表

| 问题模式分类 | 模式编号 | 复检方法 |
|-------------|---------|---------|
| 模糊词汇类 | P-01 ~ P-05 | 在新版 PRD 对应位置搜索模糊词（快、好、优化、提升、改善、加强），检查是否已替换为量化描述 |
| 结构缺失类 | P-06 ~ P-10 | 检查新版 PRD 是否补充了验收标准、异常流程、边界条件、权限定义、版本兼容说明 |
| 不一致类 | P-11 ~ P-14 | 全文搜索术语一致性、枚举值一致性、数据流向闭环、数字一致性 |
| 优先级类 | P-15 ~ P-17 | 检查优先级标注比例、核心功能优先级是否合理 |
| NESMA 合规类 | P-18 ~ P-24 | 搜索敏感词（是否、逻辑、判断、解析、校验、优化），检查是否替换为动宾结构 |
| 可行性类 | P-25 ~ P-27 | 检查依赖标注、技术预研说明、数据迁移方案 |
| 边界场景类 | P-28 ~ P-31 | 检查并发防重、空状态、超长截断、网络异常处理 |
| 其他 | P-32 ~ P-35 | 检查隐性假设消除、自然语言歧义、触发条件补充、重复功能合并 |

> 完整问题模式定义参见 `requirement-review/references/common-issues.md`

### 3.3 复检判定规则

**判定为 `verified` 的条件**（全部满足）：
1. 在新版 PRD 中，原文定位处的内容已发生实质性变更
2. 变更内容解决了原问题描述中指出的缺陷
3. `checklist_refs` 对应的所有检查项在新版内容上评估为 ✅ 通过
4. `pattern_refs` 对应的问题模式在新版内容中不再出现
5. 变更未在临近范围内引入同类新问题（如一个模糊词被替换，但引入了另一个）

**判定为 `rejected` 的条件**（满足任一）：
1. 原文定位处的内容未发生变更（宣称修复但实际未改）
2. 变更内容与原问题描述无关（改了其他东西但问题本身没动）
3. `checklist_refs` 中有任一检查项评估为 ❌ 未通过
4. `pattern_refs` 对应的问题模式仍然存在
5. 变更解决了原问题但引入了新问题（修复副作用）

### 3.4 复检执行顺序

按以下优先级执行复检：

1. **致命问题优先**：先复检 `severity = fatal` 的工单，确保阻塞项优先验证
2. **警告问题其次**：再复检 `severity = warning` 的工单
3. **建议问题最后**：最后复检 `severity = suggestion` 的工单

## 四、整改 SLA 建议

### 4.1 SLA 时限

| 严重等级 | 修复时限 | 说明 |
|---------|---------|------|
| 🔴 致命 | **24 小时**（自然时间） | 阻塞开发或测试的缺陷，必须紧急修复 |
| 🟡 警告 | **3 个工作日** | 可能引发返工或理解偏差的问题 |
| 💡 建议 | **下个版本 / 5 个工作日** | 改进空间，不阻塞当前开发 |

### 4.2 SLA 计算规则

```
SLA 起算点 = 工单 created_at
SLA 截止点 = resolved_at（首次 verified 时间）

SLA 状态：
- on_track：当前时间距截止时间 > 50% 缓冲
- at_risk：当前时间距截止时间 <= 50% 缓冲
- overdue：当前时间已超过截止时间
- exempt：已关闭的工单不计算 SLA
```

### 4.3 SLA 预警升级

| 条件 | 预警动作 |
|------|---------|
| 致命问题超时 4h | 在报告中标记为「SLA 超时 - 需立即关注」 |
| 致命问题超时 24h | 建议升级至项目经理，抄送技术负责人 |
| 警告问题超时 1 工作日 | 在报告中提醒责任人 |
| 反复打开 ≥ 3 次 | 建议召开专项评审，重新评估修复方案 |

## 五、回退与重新打开规则

### 5.1 驳回后的回退

- `rejected` → 立即回退至 `pending`
- `fix_iterations` 累加
- 保留驳回原因和复检发现，供下一轮修复参考

### 5.2 已关闭问题不重新打开

- `closed` 是终态，不允许重新打开
- 如需针对已关闭问题发起新的修复，创建新工单并引用旧工单 ID

### 5.3 已验证问题的重新打开

- `verified` 是终态，原则上不允许
- 如在后续版本中发现同类问题，创建新工单
- 新工单的 `source_ids` 中引旧工单 ID，标记为「复发」
