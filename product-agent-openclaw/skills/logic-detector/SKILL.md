# PRD 逻辑检测

## 概述

通过算法化形式推理（状态机构建、DAG 拓扑排序、谓词逻辑判定）验证 PRD 业务逻辑，填补 `requirement_review` Agent 启发式评审在形式化推理方面的空白。不对 PRD 做通用维度评审（完整性、清晰度、可行性等），仅验证"已写内容"是否存在逻辑矛盾。

## 何时使用

- PRD 涉及 3 个以上实体状态变化或 5 个以上步骤依赖关系
- Agent 评审一致性维度（Q5-Q6）或可测试性维度（Q7-Q8）得分 < 60
- 主 Agent 明确调度"逻辑检测"任务
- PRD 中存在明确的状态机定义、数据流描述或跨章节业务规则

**不适用：** 通用 PRD 评审（由 Agent 负责）、PRD 中不存在的内容（不做"应该补充"类评论）、非逻辑类问题（术语不统一、缺少验收标准等）。

## 四个检测引擎

### 引擎 1：状态机验证

对 PRD 中实体（订单、用户、任务、审批流）进行状态机形式化验证，详见 `references/state-machine-patterns.md`。

**检测项：**

| 检测项 | 判定条件 | 严重度 | 对应 Agent 检查项 |
|--------|---------|--------|-------------------|
| 不可达状态 | 从初始状态 BFS/DFS 不可达 | 🔴严重 | Q7 |
| 死锁状态 | 非终止态出度为 0 | 🔴严重 | Q7 |
| 缺失转换路径 | 业务流程需要的转换边不存在 | 🔴严重 | Q7 |
| 矛盾转换 | 同源状态+同触发条件→2+目标状态 | 🔴严重 | Q5 |
| 未定义回退 | 单向转换且回退在业务上合理但未定义 | 🟡警告 | Q7 |
| 转换条件模糊 | 触发条件使用模糊词汇（如"必要时"） | 🟡警告 | Q9 |

### 引擎 2：跨章节逻辑矛盾检测

将 PRD 各章节业务规则提取为规范化形式（`IF [条件] THEN [结论]`），检测 6 类矛盾，详见 `references/contradiction-patterns.md`。

| 矛盾类型 | 检测方法 | 致命条件 | 对应 Agent 检查项 |
|---------|---------|---------|-------------------|
| 条件冲突 | 谓词逻辑等价判定 | 核心流程条件互斥 | Q5 |
| 范围冲突 | 数值区间求交 | 合法输入域为空 | Q5/Q8 |
| 时序冲突 | 拓扑排序检测环 | 形成 ≥3 步循环 | Q6 |
| 权限冲突 | 权限矩阵求异 | 同角色同资源权限互斥 | Q5 |
| 数据冲突 | 字段存在性检查 | 必填字段无来源 | Q6 |
| 定义冲突 | 语义等价判定 | 核心术语两处含义互斥 | Q5 |

### 引擎 3：数据流字段级一致性校验

追踪数据全生命周期（输入→校验→转换→存储→查询→输出），检测字段级一致性问题，详见 `references/data-flow-checklist.md`。

| 检测项 | 判定条件 | 严重度 | 对应 Agent 检查项 |
|--------|---------|--------|-------------------|
| 字段变形 | 类型/格式/约束在传递中不一致 | 🔴严重 | Q6 |
| 字段缺失 | 被业务引用的字段在数据模型中不存在 | 🔴严重 | Q6 |
| 未定义来源 | 输出字段无上游来源 | 🟡警告 | Q6 |
| 输入输出不闭环 | 输入字段在输出中消失 | 🟡警告 | Q6 |
| 接口对齐不一致 | API 文档与业务描述字段不一致 | 🟡警告 | Q5 |
| 语义漂移 | 同名字段含义在传递中改变 | 🟡警告 | Q5 |

### 引擎 4：时序依赖验证

从业务流程提取步骤间依赖关系，构建 DAG 进行拓扑排序验证。

| 检测项 | 判定条件 | 严重度 | 对应 Agent 检查项 |
|--------|---------|--------|-------------------|
| 循环依赖 | DAG 中存在环，拓扑排序失败 | 🔴严重 | Q6 |
| 遗漏前置步骤 | 步骤后置条件无消费者 | 🔴严重 | Q3 |
| 幽灵依赖 | 前置条件无生产者 | 🔴严重 | Q6 |
| 可并行但串行化 | 无依赖步骤对可并行 | 💡建议 | - |
| 依赖描述模糊 | 依赖关系用"可能"等模糊词 | 🟡警告 | Q9 |

## 工作流程

1. **解析 PRD 结构** — 识别文档结构、实体清单、业务逻辑章节定位
2. **运行 4 个检测引擎** — 按需并行运行（顺序无关），每个引擎输出原始检测结果
3. **去重与分级** — 跨引擎去重，按 🔴致命 / 🟡警告 / 💡建议 分级，标注与 Agent 检查项的关联
4. **生成报告** — 按 `references/logic-report-template.md` 模板组织，含综合逻辑健康度评分

## 评分体系

| 等级 | 分数 | 含义 |
|------|------|------|
| A | 90-100 | 逻辑健康 |
| B | 70-89 | 逻辑基本健康 |
| C | 50-69 | 逻辑存在缺陷，需修复后重新检测 |
| D | 0-49 | 逻辑不健康，建议重新梳理 |

```
综合逻辑健康度 = 100 - Σ(扣分)
  🔴致命逻辑矛盾：-20 分/个
  🟡潜在逻辑风险：-8 分/个
  💡逻辑优化建议：-2 分/个
最低 0 分。
```

**一键否决规则（直接判 D 级）：**
- 状态机存在不可达状态且该状态是核心业务流程引用的状态
- 存在时序循环依赖
- 核心业务数据字段存在断裂
- 核心流程规则互斥（IF A THEN B 与 IF A THEN NOT B 同时存在）

## 与 Agent 的协作边界

| 维度 | requirement_review Agent | 本 Skill |
|------|--------------------------|----------|
| 评审方法 | 启发式模式匹配（35 种已知模式） | 算法化形式推理（状态机、DAG、谓词逻辑） |
| 术语一致性 | 检测术语不统一（"叫法不同"） | 检测定义矛盾（"两个叫法的定义互斥"） |
| 数据一致性 | 检测数据流向不闭环 | 追踪字段在每个阶段的具体变形/缺失 |
| 流程一致性 | 检测前后描述不一致 | 构建形式化模型证明矛盾存在 |
| 处理边界 | PRD 任何章节任何内容 | 仅处理已写的逻辑结构 |

## 红线

- 所有结论必须基于 PRD 原文证据，每条检测结果标注原文定位
- 不与 Agent 启发式评审结果冲突 — 你做形式化逻辑补充，不做重复评审
- PRD 中不存在的内容不评论 — 完整性评审是 Agent 的职责
- 如果 PRD 中无足够信息进行形式化验证，报告"该维度无足够信息"，不强行推断
- 不越权调度 `requirement_review` Agent 或其他 Skill
