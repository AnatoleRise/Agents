# 逻辑矛盾检测模式库

> 本库收录跨章节逻辑矛盾的 6 种典型模式、每种模式的检测算法、严重度分级和验证案例。

---

## 一、矛盾类型总览

| 编号 | 矛盾类型 | 定义 | 典型案例 | 对应问题模式 |
|------|---------|------|---------|-------------|
| C-1 | 条件冲突 | 同一前提条件在不同章节推导出互斥结论 | "支付成功后可退款" vs "退款前必须先取消支付" | P-41 |
| C-2 | 范围冲突 | 同一属性的合法取值范围在不同章节不一致 | "支持1-100件" vs "单次最多50件" | P-42 |
| C-3 | 时序冲突 | 步骤间的先后关系形成逻辑环 | A必须在B前 AND B必须在C前 AND C必须在A前 | P-43 |
| C-4 | 权限冲突 | 同一角色对同一资源的操作权限在不同章节矛盾 | "管理员可删除用户" vs "删除操作需超级管理员权限" | P-44 |
| C-5 | 数据冲突 | 同一数据项的定义在不同章节矛盾 | "用户ID为自增整数" vs "用户ID为UUID字符串" | P-45 |
| C-6 | 定义冲突 | 同一术语在不同章节的定义互斥 | "活跃用户=近7天登录" vs "活跃用户=近30天有操作" | P-46 |

## 二、C-1 条件冲突

### 2.1 定义

满足以下全部条件即判定为条件冲突：
```
存在规则 R1：IF A THEN B（位于章节 CH1）
存在规则 R2：IF A THEN NOT B（位于章节 CH2）
且 A 为同一前提条件（语义等价判定通过）
且 B 为同一结论（语义等价判定通过）
```

### 2.2 检测算法

```
步骤 1：条件规则提取
  从 PRD 各章节提取所有 IF-THEN 形式的规则
  正则模式：
    - 「如果(.+?)[，,]?(?:则|那么|就)(.+)」
    - 「(.+?)时[，,]?(.+?)」
    - 「当(.+?)(?:时|的情况下)[，,]?(.+)」
    - 「只有(.+?)才(?:能|可以|允许)(.+)」
    - 「(.+?)后[，,]?(.+)」(前件为"完成"的隐含条件)
    - 「禁止(.+?)在(.+?)时(.+)」(否定条件)

步骤 2：条件范式化
  对每个条件进行范式化处理：
  - 去除修饰词："系统自动"、"应当"、"必须"
  - 统一术语：将同一概念的不同叫法映射到规范名称（利用术语表）
  - 将否定表达式转为肯定表达式的否定形式：
    「未支付」→「NOT 已支付」
    「不允许」→「NOT 允许」

步骤 3：前提语义等价判定
  对每对规则 (R_i, R_j)：
    计算 R_i.precondition 和 R_j.precondition 的语义等价度
    判定规则：
    - 如果两个前提的字符串完全相同（去空白后）→ 等价
    - 如果两个前提的关键实体 + 关键动词 + 关键条件词完全一致 → 等价
    - 如果两个前提可以通过术语表映射为同一表达 → 等价

步骤 4：结论互斥判定
  对前提语义等价的规则对：
    判定其 conclusion 是否互斥：
    - 直接互斥：「可执行 X」vs「禁止执行 X」
    - 间接互斥：conclusion_A 的必要条件与 conclusion_B 互斥
    - 状态互斥：conclusion_A = "状态变为 S1", conclusion_B = "状态变为 S2", 且 S1 和 S2 在同一时刻互斥
```

### 2.3 严重度分级

| 级别 | 判定条件 | 示例 |
|------|---------|------|
| 🔴致命 | 矛盾的结论直接影响核心业务流程且无法通过上下文消解 | "支付成功后进入已支付状态" vs "支付成功后进入待发货状态" |
| 🟡警告 | 矛盾的结论影响辅助流程，或可通过上下文推断出正确版本 | "退货运费由用户承担" vs "质量问题退货运费由商家承担"（后者是前者的特例补充，非真正矛盾） |
| 💡建议 | 两个结论实质上不矛盾但表述方式容易引起误解 | "订单确认后通知用户" vs "订单创建后通知用户"（可能是同一个操作的不同称呼） |

### 2.4 检测案例

#### 案例 C-1-1：退款条件冲突（致命）

**PRD 原文**：
> 第 3 章退款流程："用户支付成功后，可在 7 天内申请退款。"
> 第 5 章取消流程："用户若需退款，必须先取消订单，取消后订单不可恢复。"

**矛盾分析**：
- 第 3 章隐含：已支付状态下可申请退款（退款时订单仍处于已支付状态）
- 第 5 章隐含：必须取消订单后才能退款（退款时订单处于已取消状态）
- 两个前提相同（用户要退款），但结论互斥（退款时订单是"已支付"还是"已取消"？状态不同意味着后续操作不同）

**检测过程**：
```
规则 R1 (第3章)：IF 用户要退款 AND 距支付 < 7天 THEN 可申请退款（订单状态保持已支付）
规则 R2 (第5章)：IF 用户要退款 THEN 必须先取消订单（订单状态变为已取消）
前提等价：是（"用户要退款"）
结论互斥：是（"保持已支付" vs "变为已取消"）
判定：🔴致命条件冲突
```

**修改建议**：
明确退款与取消的关系——是否退款必须先取消？还是退款是一种独立操作？两种设计对应的流程不同：
- 方案 A：退款 = 取消 + 退款，先取消再退款（需补充取消后的退款流程）
- 方案 B：退款是独立操作，不改变订单状态（需明确退款相关的状态流转）

#### 案例 C-1-2：通知触发条件冲突（警告）

**PRD 原文**：
> 第 2 章消息通知："用户下单成功后，系统发送下单成功通知。"
> 第 7 章异常处理："用户下单失败时，系统发送下单失败通知。但网络超时时不发送任何通知。"

**矛盾分析**：
- 第 7 章说"网络超时不发通知"，而网络超时可能是"下单失败"的一种情况
- 如果网络超时算下单失败，则两个规则矛盾：一个说发通知，一个说不发

**检测过程**：
```
规则 R1：IF 下单成功 THEN 发送成功通知
规则 R2：IF 下单失败 THEN 发送失败通知
规则 R3：IF 网络超时 THEN 不发送通知
前提关系：网络超时 ⊆ 下单失败（如果网络超时导致下单失败）
结论互斥：是（"发送失败通知" vs "不发送通知"）
判定：🟡警告条件冲突（取决于"网络超时"是否属于"下单失败"的范畴）
```

**修改建议**：
明确"网络超时"与"下单失败"的关系——网络超时是否归类为下单失败？如果是，是否应该发送通知？

#### 案例 C-1-3：伪冲突——表述歧义（建议）

**PRD 原文**：
> 第 1 章功能概述："管理员可以删除任何用户的评论。"
> 第 4 章权限模型："评论删除操作需要超级管理员权限。"

**矛盾分析**：
- 第 1 章说"管理员"可以删除，第 4 章说需要"超级管理员"权限
- 如果"管理员"是泛称、"超级管理员"是具体角色，则不矛盾
- 如果"管理员"和"超级管理员"是两个不同角色，则矛盾

**检测过程**：
```
规则 R1：IF 角色=管理员 THEN 可删除评论
规则 R2：IF 角色=超级管理员 THEN 可删除评论
前提关系："管理员"和"超级管理员"可能为同一角色（不矛盾）或不同角色（矛盾）
无法从上下文确定
判定：💡建议（无法确定是否是真正矛盾，需 PM 澄清术语）
```

## 三、C-2 范围冲突

### 3.1 定义

```
存在属性 ATTR 的两个取值范围定义：
  第 CH1 章：ATTR ∈ [MIN1, MAX1] 或 ATTR ∈ {v1, v2, ...}
  第 CH2 章：ATTR ∈ [MIN2, MAX2] 或 ATTR ∈ {w1, w2, ...}
且两个范围的交集为空 或 范围的语义约束不一致
```

### 3.2 检测算法

```
步骤 1：范围值提取
  从 PRD 文本中提取所有「属性 + 取值范围」的定义
  正则模式：
    - 「(.+?)(?:最多|不超过|≤|≤)(\\d+)(.+?)」
    - 「(.+?)(?:最少|不低于|≥|≥)(\\d+)(.+?)」
    - 「(.+?)(?:范围(?:为|是)|在|∈)[：:]?(\\d+)[-~至到](\\d+)(.+?)」
    - 「(.+?)支持(\\d+)[-~至到](\\d+)」
    - 「单次(.+?)(\\d+)」
    - 「(.+?)可(?:为|是)(.+?)(?:之一)?」

步骤 2：属性语义对齐
  对每对范围定义，判定其是否描述同一属性：
  - 属性名完全相同 → 同一属性
  - 属性名通过术语表映射为同一概念 → 同一属性
  - 属性有相同的测量单位 → 可能同一属性
  - 属性出现在同一实体的上下文中 → 可能同一属性

步骤 3：范围求交
  对同一属性的所有范围定义：
    数值范围：计算所有范围区间的交集
    枚举范围：计算所有枚举值集合的交集
    如果交集为空 → 标记为 🔴致命范围冲突
    如果交集非空但小于原始范围的 50% → 🟡警告
    如果交集非空但其中一个范围与另一个有包含关系但方向不一致 → 💡建议
```

### 3.3 严重度分级

| 级别 | 判定条件 | 示例 |
|------|---------|------|
| 🔴致命 | 两个范围的交集为空，不存在同时满足两个条件的合法输入 | "单次1-100件" vs "单次最多50件" → [1,50]非空，不致命。但"单次50件以上" vs "单次1-20件" → 交集为空 |
| 🟡警告 | 两个范围有交集但一个明显缩小了合法输入域，且未说明原因 | "支持1-100件" vs "单次最多50件" → 50-100件的订单无法处理 |
| 💡建议 | 范围描述粒度不一致但业务上可以兼容 | "大批量（10000+）" vs "单次最多50000件" 粒度不同但无矛盾 |

### 3.4 检测案例

#### 案例 C-2-1：上传文件大小范围冲突（致命）

**PRD 原文**：
> 第 2 章文件上传："支持上传文件，单文件大小不超过 100MB"
> 第 6 章技术约束："服务器接收文件大小限制为 50MB"

**矛盾分析**：
- 第 2 章定义合法范围：(0, 100MB]
- 第 6 章定义合法范围：(0, 50MB]
- 交集为 (0, 50MB]，非空
- 但第 2 章给出的 50-100MB 范围在第 6 章被否定

**检测过程**：
```
属性名：上传文件大小
范围 R1（第2章）：(0, 100] MB
范围 R2（第6章）：(0, 50] MB
交集：(0, 50] MB（有效，非致命）
但 R1-R2 = (50, 100] MB 为假阳性范围
用户按第2章使用50-100MB文件会上传失败
判定：🟡警告范围冲突
```

**修改建议**：
统一文件大小限制描述，或明确分场景限制（前端 100MB，后端 50MB，并在前端进行友好提示）。

#### 案例 C-2-2：数量范围交集为空（致命）

**PRD 原文**：
> 第 1 章功能说明："团购活动最少 10 人成团"
> 第 3 章业务规则："参与人数 5-8 人的为小型团购"

**矛盾分析**：
- 第 1 章说最少 10 人成团
- 第 3 章定义了 5-8 人的小型团购
- 如果"团购"和"小型团购"是同一业务的不同类型，则 5-8 人的范围无法满足"最少 10 人"的要求

**检测过程**：
```
属性名：团购参与人数下限
范围 R1：≥ 10（第1章）
范围 R2：[5, 8]（第3章）
交集：空！（[5,8] ∩ [10, ∞) = ∅）
判定：🔴致命范围冲突
```

**修改建议**：
明确"团购"和"小型团购"是否为同一业务类型。如果是，统一最小人数；如果不是，用不同术语区分。

## 四、C-3 时序冲突

### 4.1 定义

```
从 PRD 中提取步骤间的先后顺序关系：
  约束 1：A 必须在 B 之前
  约束 2：B 必须在 C 之前
  ...
  约束 n：X 必须在 A 之前
如果这些约束形成的图中存在环：
  (A → B → C → ... → A)
则判定为时序冲突。
```

### 4.2 检测算法

```
步骤 1：时序约束提取
  从 PRD 文本中提取步骤间的先后顺序条件：
  正则模式：
    - 「(.+?)(?:必须|需要|应)(?:在|先于)(.+?)(?:之前|前)」
    - 「只有(.+?)(?:完成)?(?:后|之后)(?:才能|才可以|才能|方可)(.+?)」
    - 「(.+?)(?:是)(.+?)(?:的前置条件|的前提)」
    - 「(.+?)依赖(?:于)?(.+?)」

步骤 2：步骤节点提取
  每个步骤映射为图中的一个节点
  对每个约束，添加从"必须在前的步骤"到"必须在后的步骤"的边

步骤 3：循环依赖检测（拓扑排序）
  对构建的有向图执行 Kahn 算法：
    - 计算每个节点的入度
    - 将所有入度为 0 的节点加入队列
    - 依次出队，将出队节点的所有后继节点的入度减 1
    - 如果入度变为 0，加入队列
  如果所有节点都被出队 → 无环
  如果存在未出队的节点 → 存在环

步骤 4：环定位
  对剩余节点执行 DFS + 颜色标记法（白/灰/黑）：
    如果发现回边（指向灰色节点），记录该环路
```

### 4.3 严重度分级

| 级别 | 判定条件 |
|------|---------|
| 🔴致命 | 存在时序循环（任意长度的环） |
| 🟡警告 | 存在隐含循环（通过中间状态的传递依赖形成环） |
| 💡建议 | 约束关系在表述上暗示循环但实际业务不存在（假阳性） |

### 4.4 检测案例

#### 案例 C-3-1：三步骤循环依赖（致命）

**PRD 原文**：
> 第 2 章数据准备："数据集必须先完成标注，才能用于模型训练。"
> 第 3 章标注流程："标注任务必须基于训练结果，才能开始质量评估标注。"
> 第 5 章训练流程："模型训练依赖质量评估标注的数据集。"

**矛盾分析**：
- 标注 → 训练（第 2 章）
- 训练 → 标注（第 3 章："基于训练结果"意味着训练必须在标注之前 → 训练 → 标注）
- 标注 → 训练（第 5 章："依赖质量评估标注的数据集"意味着标注在训练之前）

这形成了循环：标注必须先在训练之前（第2章和第5章），但标注又必须在训练之后（第3章）。

**检测过程**：
```
步骤节点：{标注, 训练}
时序约束：
  C1（第2章）：标注 → 训练
  C2（第3章）：训练 → 标注
  C3（第5章）：标注 → 训练

有向图：标注 ⇄ 训练（存在环）
判定：🔴致命时序冲突
```

**修改建议**：
明确标注和训练的迭代关系——是一轮标注→一轮训练→下一轮标注？还是一批次标注→一批次训练？需区分"初始标注"和"基于训练反馈的评估标注"。

#### 案例 C-3-2：四步骤隐含循环（警告）

**PRD 原文**：
> - 用户提交订单后才能选择支付方式
> - 选择支付方式后才能进入支付页面
> - 支付成功后系统创建发货单
> - 用户在支付页面可以返回修改订单（修改订单后重新提交订单）

**矛盾分析**：
看似正常流程：提交订单 → 选择支付方式 → 进入支付页面 → 支付 → 创建发货单
但"支付页面可以返回修改订单"隐含了：支付页面 → 修改订单 → 重新提交订单 → 选择支付方式 → 进入支付页面
形成环路：提交订单 → 选择支付方式 → 支付页面 → 修改订单 → 提交订单 → ...

这是一个预期的循环（允许用户反复修改），但如果用户无限循环修改怎么办？需要定义循环退出条件。

**检测过程**：
```
存在环：(提交订单 → 选择支付方式 → 支付页面 → 返回修改 → 提交订单)
判定：🟡警告 —— 存在环但为业务预期的交互循环
附加问题：未定义循环退出条件（如修改次数上限、超时策略）
```

## 五、C-4 权限冲突

### 5.1 定义

```
存在角色 R 和操作 OP：
  第 CH1 章：角色 R 可以执行操作 OP
  第 CH2 章：角色 R 不可以执行操作 OP（或操作 OP 需要角色 R' 的权限，且 R ≠ R'）
```

### 5.2 检测算法

```
步骤 1：构建权限矩阵
  从 PRD 各章节提取「角色-操作」对：
  矩阵 M[角色][操作] = {'allow', 'deny', 'undefined'}

步骤 2：冲突检测
  对每个 (角色, 操作) 对：
    如果 M[角色][操作] 存在两个不同的值 → 权限冲突
    如果 M[角色_泛称][操作] = 'allow' 且 M[角色_具体][操作] = 'deny'
       且 角色_具体 是 角色_泛称 的子类型 → 权限冲突
```

### 5.3 检测案例

#### 案例 C-4-1：管理员权限矛盾（致命）

**PRD 原文**：
> 第 3 章用户管理："管理员可以删除任意用户账户。"
> 第 8 章安全规范："用户删除操作仅在用户本人发起注销申请并经超级管理员审批后方可执行。管理员只可冻结用户，不可直接删除。"

**矛盾分析**：
- 第 3 章：M[管理员][删除用户] = 'allow'
- 第 8 章：M[管理员][删除用户] = 'deny'

**判定**：🔴致命权限冲突

**修改建议**：
明确管理员的删除权限——管理员到底能不能直接删除用户？如果不能，第 3 章需要修正。同时区分"管理员"和"超级管理员"的权限边界。

## 六、C-5 数据冲突

### 6.1 定义

```
存在数据项 D：
  第 CH1 章定义 D 的类型/格式/约束为 T1
  第 CH2 章定义 D 的类型/格式/约束为 T2
  且 T1 和 T2 在技术上不兼容
```

### 6.2 技术不兼容判定表

| 属性 | 不兼容示例 |
|------|-----------|
| 数据类型 | INT vs VARCHAR / 整数 vs 字符串 |
| 数据长度 | CHAR(10) vs CHAR(50)（长度不一致视为警告） |
| 必填性 | 必填 vs 可选 |
| 唯一性 | 全局唯一 vs 允许重复 |
| 枚举值 | 枚举集合 {A,B} vs {A,C} 且无 B 或 C 的迁移说明 |

### 6.3 检测案例

#### 案例 C-5-1：用户ID类型冲突（致命）

**PRD 原文**：
> 第 3 章数据模型："用户 ID 为自增整数，从 10000 开始"
> 第 6 章接口定义："user_id 字段为 UUID v4 格式字符串"

**矛盾分析**：
同一字段 `user_id` 在两个章节中有不兼容的类型定义。

**判定**：🔴致命数据冲突

**修改建议**：
统一 user_id 的数据类型。如果是历史遗留（自增整数 → UUID 迁移），需说明迁移策略。

## 七、C-6 定义冲突

### 7.1 定义

```
存在术语 T：
  第 CH1 章定义 T 的含义为 M1
  第 CH2 章定义 T 的含义为 M2
  且 M1 和 M2 导致不同的业务决策或数据范围
```

### 7.2 检测方法

语义等价需要更细致的判定——不同于术语不统一（同一个概念多个叫名），定义冲突是"同一个术语被赋了两个互斥的含义"。

### 7.3 检测案例

#### 案例 C-6-1：活跃用户定义冲突（警告）

**PRD 原文**：
> 第 1 章指标定义："活跃用户指近 7 天内有登录行为的用户。"
> 第 7 章数据看板："活跃用户口径：近 30 天内有操作行为的用户。"

**矛盾分析**：
- 定义 1：时间窗口 7 天 + 行为 登录
- 定义 2：时间窗口 30 天 + 行为 操作
- 这两个定义对哪些用户算"活跃"产生完全不同的结果

**判定**：🟡警告定义冲突

**修改建议**：
区分"活跃用户"和"月活用户"两个概念，或统一口径。注意这与 P-11（术语不统一）的区别：P-11 是"同一概念叫法不同"，而此处是"同一叫法含义不同"——后者更危险。

## 八、矛盾检测优先级矩阵

在运行跨章节矛盾检测时，按以下优先级加载规则对进行比较：

| 优先级 | 比较范围 | 说明 |
|--------|---------|------|
| 1 | 同一实体 + 同一属性 | 优先比较同一实体同一属性的规则（最可能发现致命矛盾） |
| 2 | 同一实体 + 相关属性 | 比较同一实体的不同属性之间的隐含矛盾 |
| 3 | 不同实体 + 直接引用 | 比较通过引用关系关联的不同实体间的规则 |
| 4 | 不同实体 + 语义关联 | 通过术语表推断的不同实体间的规则 |

---

## 九、综合检测伪代码

```python
class ContradictionDetector:
    """跨章节逻辑矛盾检测器"""

    def extract_rules(self, prd_text: str) -> list[Rule]:
        """从 PRD 各章节提取规则"""
        rules = []
        chapters = self.split_into_chapters(prd_text)
        for ch_name, ch_text in chapters.items():
            # 提取 IF-THEN 规则
            if_then = self.extract_if_then(ch_text)
            # 提取范围约束
            ranges = self.extract_range_constraints(ch_text)
            # 提取时序约束
            temporal = self.extract_temporal_constraints(ch_text)
            # 提取权限规则
            permissions = self.extract_permission_rules(ch_text)
            # 提取数据定义
            data_defs = self.extract_data_definitions(ch_text)
            for r in if_then + ranges + temporal + permissions + data_defs:
                r.chapter = ch_name
                rules.append(r)
        return rules

    def detect_all_contradictions(self, rules) -> list[Contradiction]:
        """运行全部 6 种矛盾检测"""
        results = []
        results.extend(self.detect_condition_conflicts(rules))  # C-1
        results.extend(self.detect_range_conflicts(rules))      # C-2
        results.extend(self.detect_temporal_conflicts(rules))   # C-3
        results.extend(self.detect_permission_conflicts(rules)) # C-4
        results.extend(self.detect_data_conflicts(rules))       # C-5
        results.extend(self.detect_definition_conflicts(rules)) # C-6
        return self.deduplicate(results)

    def semantic_equivalence(self, expr1: str, expr2: str, glossary: dict) -> bool:
        """判断两个表达是否语义等价（使用术语表）"""
        # 1. 字符串完全相同 → 等价
        if expr1.strip() == expr2.strip():
            return True
        # 2. 去停用词后完全相同 → 等价
        if self.normalize(expr1) == self.normalize(expr2):
            return True
        # 3. 使用术语表映射后完全相同 → 等价
        if self.map_terms(expr1, glossary) == self.map_terms(expr2, glossary):
            return True
        # 4. 去掉否定词后等价且一个是否定形式 → 不认为是等价（这是互斥条件）
        return False

    def is_mutually_exclusive(self, conclusion1: str, conclusion2: str) -> bool:
        """判断两个结论是否互斥"""
        # 1. 直接否定：「可执行 X」vs「不可执行 X」
        # 2. 状态互斥：指向不同且互斥的目标状态
        # 3. 值互斥：赋值不同的且不可同时成立的值

    def detect_temporal_cycles(self, constraints) -> list[list[str]]:
        """拓扑排序检测时序环"""
        graph = self.build_directed_graph(constraints)
        in_degree = {node: 0 for node in graph}
        for src in graph:
            for dst in graph[src]:
                in_degree[dst] += 1
        queue = [n for n, d in in_degree.items() if d == 0]
        sorted_nodes = []
        while queue:
            node = queue.pop(0)
            sorted_nodes.append(node)
            for neighbor in graph[node]:
                in_degree[neighbor] -= 1
                if in_degree[neighbor] == 0:
                    queue.append(neighbor)
        remaining = [n for n in graph if n not in sorted_nodes]
        return self.find_cycles(remaining, graph) if remaining else []

    def grade_severity(self, contradiction: Contradiction) -> str:
        """对矛盾进行严重度分级"""
        # 致命：导致业务流程不可执行或存在逻辑死锁
        if contradiction.affects_core_flow():
            return '🔴致命'
        # 警告：导致不同角色可能做出互斥决策
        if contradiction.causes_ambiguity():
            return '🟡警告'
        # 建议：不真正矛盾但表述容易误解
        return '💡建议'
```
